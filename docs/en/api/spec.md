# API Specification

This document describes the current request, response, and stream envelope shapes defined in `ss-protocol`. It focuses on the wire structure rather than backend implementation details.

## 1. Overview

- Requests use a JSON-RPC 2.0-style envelope.
- Unary responses use a single JSON-RPC response.
- Streaming responses use a separate server event envelope rather than incremental JSON-RPC `result` messages.
- The protocol is transport-agnostic and can be carried over HTTP, WebSocket, or other channels.

## 2. Request Shape

Request message:

```json
{
  "jsonrpc": "2.0",
  "id": "req-1",
  "session_id": "session-1",
  "method": "story.generate",
  "params": {}
}
```

Field meanings:

- `jsonrpc`: always `"2.0"`.
- `id`: request identifier generated by the client and echoed by the server.
- `session_id`: optional. Required for session-bound methods only.
- `method`: request method name.
- `params`: method-specific parameter object. Defaults to an empty object when omitted.

General rules:

- Request parameter objects are closed shapes; undeclared fields should be treated as invalid params.
- Character cards, resources, stories, and sessions are referenced by id.
- `session_id` lives at the top level and is not duplicated inside session method params.

## 3. Response Shape

Successful response:

```json
{
  "jsonrpc": "2.0",
  "id": "req-1",
  "session_id": "session-1",
  "result": {
    "type": "story_generated"
  }
}
```

Error response:

```json
{
  "jsonrpc": "2.0",
  "id": "req-1",
  "error": {
    "code": -32602,
    "message": "invalid params"
  }
}
```

Rules:

- A response must contain either `result` or `error`.
- `result` and `error` cannot appear together.
- `session_id` is echoed on session-bound responses.

### 3.1 Error Shape

```json
{
  "code": 40909,
  "message": "story has sessions",
  "data": {}
}
```

Fields:

- `code`: numeric error code.
- `message`: readable error text.
- `data`: optional extra payload.

Current error code mapping:

| Name | Value |
| --- | ---: |
| `parse_error` | `-32700` |
| `invalid_request` | `-32600` |
| `method_not_found` | `-32601` |
| `invalid_params` | `-32602` |
| `internal_error` | `-32603` |
| `not_found` | `40404` |
| `conflict` | `40909` |
| `backend_error` | `50001` |
| `stream_error` | `50002` |

## 4. Stream Shape

Streaming responses use a separate server event envelope:

```json
{
  "message_type": "stream",
  "request_id": "req-1",
  "session_id": "session-1",
  "sequence": 0,
  "frame": {
    "type": "started"
  }
}
```

Fields:

- `message_type`: currently always `"stream"`.
- `request_id`: the request id that opened the stream.
- `session_id`: optional, usually present for session-bound streams.
- `sequence`: per-stream sequence number starting from `0`.
- `frame`: stream frame payload.

### 4.1 StreamFrame Shape

There are four current frame shapes:

```json
{ "type": "started" }
```

```json
{
  "type": "event",
  "body": {
    "type": "narrator_text_delta"
  }
}
```

```json
{
  "type": "completed",
  "response": {
    "type": "turn_completed"
  }
}
```

```json
{
  "type": "failed",
  "error": {
    "code": 50002,
    "message": "turn failed"
  }
}
```

Stream rules:

- A stream must begin with `started`.
- It may emit any number of `event` frames.
- It must finish with either `completed` or `failed`.
- `completed` must contain the final aggregated `response`, so the client does not need to reconstruct the final state from deltas.

## 5. Method Families

Current methods are grouped as:

- `upload.*`: chunked upload
- `character.*`: character card object access and deletion
- `story_resources.*`: story generation resource objects
- `story.*`: planning, generation, retrieval, deletion, and session start
- `session.*`: session retrieval, execution, update, and deletion
- `config.*`: global configuration access and update

## 6. Session Binding Rules

These methods require a top-level `session_id`:

- `session.get`
- `session.delete`
- `session.run_turn`
- `session.update_player_description`
- `session.get_runtime_snapshot`
- `session.get_config`
- `session.update_config`

These methods do not require `session_id`:

- `upload.*`
- `character.*`
- `story_resources.*`
- `story.*`
- `session.list`
- `config.*`

## 7. Chunked Upload Protocol

Large files use a three-step flow:

1. `upload.init`
2. `upload.chunk` repeated
3. `upload.complete`

Upload rules:

- The current `target_kind` only supports `character_card`.
- Chunks are sent through `payload_base64`.
- Every chunk carries both `chunk_index` and `offset`.
- After completion, the server parses the `.chr` archive and stores it as a character object.

## 8. `.chr` Character Archive Format

`.chr` is a ZIP container with these fixed entries:

- `manifest.json`
- `content.json`
- `cover.<ext>`

### 8.1 manifest.json

```json
{
  "format": "sillystage_character_card",
  "version": 1,
  "character_id": "merchant",
  "content_path": "content.json",
  "cover_path": "cover.png",
  "cover_mime_type": "image/png"
}
```

### 8.2 content.json

```json
{
  "id": "merchant",
  "name": "Old Merchant",
  "personality": "greedy but friendly trader",
  "style": "talkative, casual, slightly cunning",
  "tendencies": ["likes profitable deals"],
  "state_schema": {},
  "system_prompt": "..."
}
```

### 8.3 cover

Supported MIME types:

- `image/png`
- `image/jpeg`
- `image/webp`

## 10. Runtime Streaming Return Pattern

`session.run_turn` is currently the only standard streaming method:

1. A unary JSON-RPC ack returns `turn_stream_accepted`
2. The server then emits stream events
3. `completed.response` carries `turn_completed`
4. Or the stream ends with `failed.error`

Clients should treat the unary ack and the following stream as two parts of the same request lifecycle.
